# Tic‑Tac‑Toe Web Game – Development Plan

## Project Overview
Create a **single‑page web application** that allows two players (human vs human) to play classic Tic‑Tac‑Toe in a browser.  
The game must provide:

1. A 3×3 grid UI.
2. Ability for players to click cells to place their mark (`X` or `O`).
3. Automatic turn switching.
4. Real‑time win/draw detection with visual feedback (highlight winning line, display winner or draw message).
5. Reset button to start a new game.

The project will be built using plain **HTML, CSS, and vanilla JavaScript** for maximum compatibility and simplicity. No build tooling is required; the code can run by opening `index.html` locally.

## Detailed Feature Specification

| Feature | Description | Implementation Notes |
|---------|-------------|----------------------|
| **UI Layout** | 3×3 grid of clickable cells, a status bar, and a reset button. | Use CSS Grid for the board (`display: grid; grid-template-columns: repeat(3,1fr);`). Each cell is a `<button>` with data attributes `data-row` & `data-col`. |
| **Turn Management** | Alternate between Player X (first) and Player O. Show current player in status bar. | Keep a boolean `isXTurn`; toggle after each valid move. |
| **Move Validation** | Prevent clicking on an occupied cell or playing after game over. | On click, check if the cell's text content is empty and that `gameOver` flag is false. |
| **Win Detection** | Detect any row, column, or diagonal containing three identical marks. Highlight winning cells. | After each move, call `checkWinner()` which scans rows, columns, diagonals. If win: set `gameOver = true`, update status bar, add a CSS class (`win`) to the three cells. |
| **Draw Detection** | When all 9 cells are filled without a winner. | Count moves; if move count === 9 and no winner, declare draw. |
| **Reset Game** | Clear board, reset state variables, hide win/draw message. | On clicking “Reset” button, clear cell contents, remove `win` classes, set `isXTurn = true`, `gameOver = false`. |
| **Accessibility** | Use semantic tags (`<button>`), ARIA roles if needed, and focus styles. | Add `aria-label="Player X's turn"` to status bar; ensure buttons are reachable via keyboard. |

## Implementation Steps (Priority Order)

1. **Setup Project Skeleton**
   - Create `src/index.html`, `src/styles.css`, `src/script.js`.
   - Link CSS & JS in HTML.

2. **Build the Board UI**
   - Generate 9 button elements inside a container.
   - Apply grid layout via CSS.
   - Add data attributes for row/col indices.

3. **Implement Core Game Logic**
   - Initialize state variables (`board`, `isXTurn`, `gameOver`).
   - Attach click event listeners to cells.
   - Update cell text and state on valid moves.

4. **Add Turn Switching & Status Display**
   - Toggle `isXTurn` after each move.
   - Update status bar with current player.

5. **Implement Win Detection Algorithm**
   - Write helper functions: `checkRows`, `checkColumns`, `checkDiagonals`.
   - On win, set `gameOver = true`, highlight cells, update status bar to show winner.

6. **Add Draw Detection**
   - After each move, if no win and board full → declare draw, set `gameOver`.

7. **Implement Reset Functionality**
   - Attach click handler to “Reset” button.
   - Clear board and reset all state variables.

8. **Styling Enhancements**
   - Add hover/focus styles for cells.
   - Highlight winning line (e.g., green border or background).
   - Style status bar and reset button.

9. **Accessibility & Edge Cases**
   - Ensure keyboard navigation works.
   - Test on mobile browsers.
   - Handle rapid clicks gracefully.

10. **Final QA & Documentation**
    - Verify win/draw detection in all scenarios.
    - Add comments to JS for maintainability.
    - Write a brief README with usage instructions (optional).
